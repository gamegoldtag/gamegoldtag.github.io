API 캐시 전략: 응답 속도와 데이터 신선도를 함께 잡는 설계

게시일: 2025년 12월 18일 · 11분 읽기

API 성능 이슈에서 가장 흔한 실수는 캐시를 단순히 "켜는 것"입니다. 캐시는 설계가 절반이며, 특히 만료 정책과 무효화 전략이 없으면 데이터 신뢰도를 빠르게 잃습니다.

핵심 원칙

조회 빈도가 높고 변경 빈도가 낮은 엔드포인트부터 적용합니다. 목록 캐시와 상세 캐시를 분리하고, 키 구성에는 버전과 사용자 스코프를 명확히 포함해야 합니다.

TTL과 무효화

TTL 하나로 모든 문제를 풀 수 없습니다. 쓰기 이벤트가 발생하면 관련 캐시를 명시적으로 무효화하고, 그 외 경로는 TTL로 보완하는 하이브리드 방식이 안정적입니다.

운영 지표

hit ratio, stale read rate, p95 latency를 함께 모니터링하면 캐시가 실제로 가치 있는지 판단할 수 있습니다. hit ratio가 높아도 stale read가 높으면 사용자 경험은 나빠질 수 있습니다.

캐시는 성능 기능이 아니라 데이터 계약의 일부입니다. 계약을 먼저 정의하면 장애 가능성이 크게 줄어듭니다.
