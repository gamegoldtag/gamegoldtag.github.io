E2E 테스트 flaky 줄이기: 실패 원인을 구조적으로 제거하는 방법

게시일: 2025년 12월 6일 · 10분 읽기

E2E가 불안정하면 테스트가 신뢰를 잃고, 결국 파이프라인에서 꺼지게 됩니다. flaky는 테스트 품질 문제가 아니라 시스템 경계 관리 문제로 보는 것이 맞습니다.

자주 깨지는 원인

비동기 대기 타이밍, 외부 API 응답 편차, 공유 테스트 데이터 충돌이 핵심 원인입니다. sleep 기반 대기는 제거하고, 상태 기반 대기 조건으로 바꿔야 합니다.

안정화 전략

테스트 계층을 smoke/regression/full로 분리하고, PR 단계에는 짧고 안정적인 세트만 실행합니다. 외부 의존성은 가능한 한 mock server로 고정해 변동성을 줄입니다.

실패 분석 루프

실패 시 스크린샷만 저장하면 부족합니다. 콘솔 로그, 네트워크 로그, 테스트 이벤트 타임라인을 함께 저장해야 재현 속도가 빨라집니다.

E2E의 목표는 커버리지 최대화가 아니라 배포 리스크 최소화입니다. 목적에 맞게 테스트를 줄이고 정리하는 것이 더 중요합니다.
